## 設計詳細

### ポーリング仕様
- 間隔: **30 分** (POLL_INTERVAL_MS = 1,800,000 ms)
- 初回: アプリ起動 3 秒後に即時実行
- 1 回あたり上限: **5 件** (API コスト考慮)
- 対象: 未読 (`isUnread = true`) かつ 未処理 (localStorage に未登録)

### localStorage スキーマ (`core_shadow_drafts_v1`)
```json
[
  {
    "messageId": "Gmail Message ID",
    "threadId": "Gmail Thread ID",
    "subject": "件名",
    "from": "差出人 <email>",
    "draftText": "AI 生成返信下書き本文",
    "importance": "high | mid | low | spam",
    "summary": "1-2 行要約",
    "generatedAt": "ISO8601",
    "sent": false,
    "dismissed": false
  }
]
```
- 最大保存件数: 50 件
- 送信済み (`sent: true`) / 無視済み (`dismissed: true`) はアクティブリストから除外

### プライバシー保護
- 返信下書きは **localStorage のみ** に保存
- Gmail アクセストークンは CORE のサーバに送信しない
- 送信はユーザーが「✉️ 送信」ボタンを明示的に押した場合のみ実行

## UI フロー

```
Gmail 接続済み
  → useShadowSecretary が 30 分おきにバックグラウンドで fetchInbox()
  → 未読メールを triageEmails() に渡し needsReply=true なもの下書き生成
  → TodayBrief に「📬 返信下書き済 N件」バッジ表示
  → バッジ / QuickActions の「📬 下書き済み」ボタン → ShadowSecretaryPanel 開く
  → ユーザーが確認・修正 → 送信 or 無視
```

## 仮想テストシナリオ

### シナリオ A: 正常ポーリング
1. Gmail 接続済み状態でアプリ起動
2. 3 秒後に `fetchInbox(20)` → 未読 3 件あり
3. `triageEmails` が 3 件分析し、うち 2 件が `needsReply=true`
4. localStorage に 2 件の下書きが保存される
5. TodayBrief に「📬 返信下書き済 2件」バッジ表示
6. バッジタップ → ShadowSecretaryPanel が開く

### シナリオ B: Gmail 未接続時
1. Gmail 未接続でアプリ起動
2. `isGmailConnected()` = false → ポーリング開始せず
3. TodayBrief にバッジ表示なし (shadowDraftCount = 0)
4. QuickActions の「📬 下書き済み」から空パネルを開ける (正常動作)

### シナリオ C: 同一メールの重複防止
1. 初回ポーリングで messageId="abc" の下書きを生成
2. 30 分後の 2 回目ポーリング時、"abc" は knownIds に含まれる
3. "abc" は再処理されない

### シナリオ D: 送信フロー
1. ShadowSecretaryPanel でメールを展開
2. 「✎ 修正」ボタンで本文を編集
3. 「✉️ 送信」ボタンタップ → `sendReply()` が Gmail REST API にポスト
4. 下書きが `sent: true` になりアクティブリストから消える

### シナリオ E: API キー未設定
1. `settings.claudeApiKey` も `VITE_CLAUDE_API_KEY` も未設定
2. `runPoll()` が早期リターン → エラーなし
3. ユーザー体験に影響なし

## 既知の制限

- Gmail トークンの有効期限 (通常 1 時間) 切れ時はポーリングが失敗し次回まで待機
- ポーリング中にアプリをリロードすると 3 秒後に再実行される
- 大量未読時は 5 件上限により全件処理に複数サイクルかかる
